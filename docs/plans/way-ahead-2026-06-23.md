# 2. The thesis behind the plan

**Reframe (vs my previous answer):** The next 10× is not in adding mechanism (it's mostly built) nor in adding doctrine (it's mature). The next 10× is in **using what was just shipped, on real repos, with verifiable output, and turning that into evidence the outside world can read.**

Three asymmetries drive the ordering:

1. **Substrate exists but has zero field hours.** The 4-layer substrate is the most defensible thing in the repo. Every day it sits unused is a day it can't be cited. The fastest path to credibility is one good, end-to-end, archived adversarial run on a known-vulnerable public repo, with Layer 1/2/3/4 all engaged.

2. **Twelve missions, two proven.** Surface area is a tax. Every unproven mission costs maintenance, dilutes the marketing story, and creates ambiguity about what the product is. The exploratory-promotion rule already exists; apply it.

3. **Method is the moat; nobody publishes their method.** Other frameworks ship code. We ship doctrine + enforcement. The proof of method is a benchmark + a run-archive that anyone can inspect. The proof is the artifact, not the README.

# Ordering principle

Each commit must do ONE of:
- **Generate an artifact** (a benchmark number, a run archive, a published trace) that the outside world can cite.
- **Close a real, named gap** (Gap 1 from landscape research, Layer 3 blind-fix guard, one borrowable pattern).
- **Reduce surface area** (demote unproven missions, retire dead paths).

No commit may **add doctrine** unless it also ships the enforcing test. No commit may **add a new mission** until ≥10 of the existing 12 have run-archive evidence on an external repo. No commit may **add UI** before a structured trace stream exists.

# Commit B — Layer 3 (blind-fix) mechanical guard

**Goal:** Layer 3 is the only one of the 4 substrate layers without a Python verifier. Ship one.

**Why second:** Commit A will produce blind-fix attestations as a side effect. Before we trust them, we need a script that can fail them when a fixer agent cheated. Right now `blind_fix_attested: true` is a vibe.

**Concrete deliverables:**
- `skills/autonomous-fleet-core/references/blind-fix.md` (new) — formal spec of the protocol, with the verifier's invariants:
  - For each closed finding, two distinct git refs must exist: `reviewer-quote@<sha>` (when the reviewer wrote the `quoted_line`) and `fixer-draft@<sha>` (when the fixer first proposed its diff).
  - The fixer's draft commit must NOT contain the reviewer's `quoted_line` verbatim in its message or diff context unless the line itself was the bug.
  - The fixer-draft sha must be earlier (by mtime, by commit timestamp, and by ledger entry) than any commit that references the reviewer findings file.
- `scripts/verify_blind_fix.py` (new) — CLI: `verify_blind_fix.py .fleet/runs/<run_id>/` returns exit 0 if every closed finding has a valid blind-fix chain.
- `assets/fleet-review-findings.schema.json` — extend with optional `blind_fix_chain: {reviewer_quote_sha, fixer_draft_sha, integration_sha}` block.
- `tests/test_verify_blind_fix.py` (new) — 8–10 cases: happy path, fixer-cheated (quoted_line in draft), out-of-order timestamps, ledger inconsistency, missing draft sha, etc.
- Wire it into `scripts/validate-all.sh` as a new step (`validate-blind-fix`).

**What this commit changes:**
- +1 reference doc
- +1 script
- +1 test file
- +1 line in `validate-all.sh`
- +1 line in CI workflow
- +1 entry in README.md "Individual validators" list (we just updated this; adding one row is trivial)
- Net: Layer 3 has parity with Layers 1, 2, 4. Doctrine and enforcement now match.

**Effort:** 2 days.

**Done means:** All four layers have a Python verifier, a test file, an asset (where applicable), and a `validate-all.sh` step. `tests/test_verify_blind_fix.py` runs in CI. The run-archive from Commit A re-validates with blind-fix included.

# Commit C — External adversarial benchmark (the artifact that defends the methodology)

**Goal:** Run `adversarial-review-and-fix` on 5 known-vulnerable or known-buggy public OSS repos. Publish numbers.

**Why third:** Commits A+B make the substrate trustworthy. Now we need an artifact that proves the substrate adds value externally, not just internally on a repo whose findings we already know.

**Falsifiable comparator (RESOLVED 2026-06-23 PM):** Each bench target runs TWICE — once **substrate-off** (no stop-verify hook, no `verify_findings` schema enforcement, no blind-fix attestation) and once **substrate-on** (all four layers active). The headline metric is the delta:
- Substrate-on findings precision (fraction of findings with `verified: true`) MINUS substrate-off precision
- Substrate-on close rate after blind-fix MINUS substrate-off close rate
- Substrate-on cost per closed-and-verified finding MINUS substrate-off cost per closed finding

This is the falsifiable claim the substrate must defend: "turning on the 4 layers raises precision by X% and lowers wasted spend by $Y per finding." If the delta is small or negative, that's the substrate's fault, not the methodology's. We learn either way.

**Target selection (5 repos, sized to fit a single batch):**

| Repo | Why | Known issue surface |
|---|---|---|
| `pallets/click` (1k stars equivalent) | Mature, widely-imported Python CLI lib; minor smell density expected | Type narrowness, error paths |
| `python-jsonschema/jsonschema` | We use it. A finding here is dogfood-grade. | Performance, edge cases |
| One repo from `bandit` test corpus | Designed to have real vulnerabilities | Crypto misuse, command injection |
| One repo from SWE-Bench Lite (a single bug-fix instance) | Externally-validated correctness; we can compare to SWE-Bench leaderboard | Reproducible defect |
| `github/gemoji` | Already in our dogfood; comparison baseline | Doc + license drift |

**Metrics to publish (per repo):**
- Findings found
- Findings verified (`verify_findings.py` passed)
- Findings closed via fix (PR landed or attestable draft)
- Findings withdrawn after blind-fix challenged them
- Total cost estimate (`fleet-outcome.cost_estimate`)
- Wall clock to FREEZE; wall clock to all-closed
- Stop-verify activations (how often did Layer 2 block a premature stop)

**Concrete deliverables:**
- `docs/external-dogfood/adversarial-bench-2026-06.md` (new) — methodology, results table, links to each `.fleet/runs/` archive
- `docs/external-dogfood/adversarial-bench/` (new dir) — one `<repo-slug>.md` per target with the full archive listing
- `scripts/bench-adversarial.sh` (new) — driver script that takes a list of repos and runs the mission with archiving, with one of the supported adapters
- `tests/test_bench_adversarial.py` (new) — asserts the bench driver produces a manifest-conforming archive on a fixture repo (test repo with 2 known issues planted; CI runs this against a checked-in fixture, not real network)
- A short post: `docs/external-dogfood/adversarial-bench-summary.md` (the public-facing version, with the "here are the numbers" framing)

**What this commit changes:**
- 1 new driver script
- 1 new test file (against a fixture, so CI stays deterministic and offline)
- 1 new dogfood doc + sub-dir with N archive descriptions
- 0 new doctrine
- 0 new missions
- Net: First public artifact that says *"the methodology produces these numbers."*

**Effort:** 3–4 days (the runs take real wall-clock time; the driver is small).

**Pre-conditions:**
- Commit A must be merged (run-archive proven)
- Commit B must be merged (blind-fix guard active, so attestations are real)
