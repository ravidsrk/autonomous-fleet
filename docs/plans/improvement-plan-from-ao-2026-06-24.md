# Plan — autonomous-fleet improvements from the Agent Orchestrator comparison

Author: deep-dive comparison session, 2026-06-24.

Status: **PROPOSAL** — derived from a deep technical dig of ComposioHQ Agent Orchestrator
(AO) against autonomous-fleet (AF). AO is a built TypeScript product with a daemon control
plane + dashboard; AF is a no-daemon, tool-agnostic prose engine + python validators. Every
item below RE-EXPRESSES an AO runtime mechanism as an AF-native checkable invariant (a
scripts/lib validator, a one-shot script, an engine.md doctrine block, or a mutation entry).
No daemon, no UI service.

## Headline

12 confirmed items in 3 waves by leverage-per-effort. Wave 1: four low-effort wins (SHA-pin validator, watch dashboard, ledger-contradiction guard, trace health roll-up) that reuse the established skeleton: argparse CLI + lib + substrate_disable kill-switch as first action + a nullglob loop entry in validate-all.sh + one mutations.yaml entry. Wave 2: the medium scanners and extensibility-contract validators. Wave 3: structural schema-version bumps and engine.md splits. Every validator is a one-shot stateless python lib + thin CLI; no daemon, no service, no port. CORRECTION to the brief: the AF mutation gate (scripts/mutation_check.py) asserts a mutation is CAUGHT (guards FAIL under mutation), it does NOT assert the mutation survives. The ledger-invariants acceptance test as written ('mutation must survive, guards still pass') is backwards for this repo and would itself fail the gate; I restate every acceptance test the AF-native way (mutation -> guard goes RED). Dependencies are folded into each item's design field since the schema has no dependency slot.

## Wave 1 — low-effort high-leverage quick wins

Each reuses the existing validator skeleton 1:1 (argparse CLI + lib + substrate_disable.is_disabled/announce_disabled as the first line of main + a nullglob loop entry in validate-all.sh + one mutations.yaml entry). They turn the four cheapest prose disciplines into checkable invariants and add a live dashboard affordance, with no schema bumps and no new external-call surface beyond the single git rev-parse pattern verify_blind_fix already established. Do these first: highest caught-defect per line, and they bed in the pattern the Wave 2 scanners reuse.

### SHA-PIN enforcement validator: flag REVIEWED as OUTDATED when branch HEAD diverges from the reviewed SHA

- **Effort:** medium (sits in Wave 1 because it is the prime enforce-the-prose candidate and reuses the skeleton verbatim; the only novelty is the post-merge N/A branch)
- **Files:** scripts/lib/verify_sha_pin.py, scripts/verify_sha_pin.py, skills/autonomous-fleet-core/assets/fleet-sha-pin.schema.json, scripts/validate-all.sh (nullglob loop entry next to verify-blind-fix), skills/autonomous-fleet-core/references/engine.md (reviewer emits sha-pin.json cross-ref, near lines 754-758), tests/test_sha_pin.py, tests/mutations.yaml
- **Design:** New scripts/lib/verify_sha_pin.py + thin scripts/verify_sha_pin.py mirroring verify_blind_fix.py: argparse, substrate_disable.is_disabled('FLEET_DISABLE_SHA_PIN')+announce_disabled as the first line of main(), exit 0/1. Input is machine substrate, not the markdown ledger: the reviewer writes a sibling .fleet/runs/<run_id>/sha-pin.json {review_id, reviewed_sha (40-hex), branch, verdict} at PASS time. For every sha-pin.json whose verdict is approve/PASS, run `git -C <repo-root> rev-parse <branch>` (same subprocess pattern verify_blind_fix is structured around) and assert reviewed_sha == current branch HEAD; on divergence emit FAIL '<branch> moved <reviewed_sha>..<head>: REVIEWED is OUTDATED, force re-review' and exit 1. Post-merge handling (CRITICAL): only enforce for verdict=approve runs whose task is NOT yet MERGED; when git rev-parse <branch> fails with unknown-revision AND a merged marker exists, treat as N/A not FAIL (otherwise it flags every shipped task whose branch was deleted). Add assets/fleet-sha-pin.schema.json (pinned 1.0) validated in the lib. Cross-ref engine.md so the reviewer emits sha-pin.json. DEPENDENCIES: none (standalone; reviewer-emits-sha-pin.json wiring is prose-only in engine.md). Establishes the git-subprocess-in-validator pattern the Wave 2 recovery scanner reuses.
- **Acceptance (fail-if-inert):** tests/test_sha_pin.py: (a) temp git repo, commit C1, sha-pin.json reviewed_sha=C1 verdict=approve, task NOT merged -> verify_sha_pin returns 0; (b) commit C2 on the same branch (HEAD moves), re-run -> returns 1 and the message names both C1 and C2; (c) branch deleted + merged marker present -> returns 0 (N/A, not FAIL). mutations.yaml id=sha-pin-divergence-ignored: in verify_sha_pin.py replace the `reviewed_sha == head` comparison with `True`, guards [tests/test_sha_pin.py] — the gate asserts case (b) goes RED, proving the check is load-bearing, not a tautology.

## What to explicitly NOT do

- Do NOT build a daemon, long-lived service, socket, PID file, or port for the dashboard. The --watch flag is a FOREGROUND poll loop that dies with the terminal (Ctrl-C); AO's live SSE/websocket Dashboard.tsx model is explicitly rejected. The unit test drives watch_once() directly so CI never sleeps.
- Do NOT rewrite the markdown ledger as a typed Zod-style three-axis state machine (AO lifecycle-state.ts). The flat boolean bag is mostly monotonic and the cross-signal contradictions are already covered by SIGNAL RECONCILIATION (engine.md 397-418) and TERMINATE. Add 3 scoped boolean invariants, not a state machine.
- Do NOT make recovery_scan.py, the orphan sweep, or the reviewer-sandbox validator EXECUTE destructive actions. They are advisory: they emit recommended actions (CLEANUP_WORKTREE / ARCHIVE_ORPHAN / ESCALATE) and the coordinator/engine decides. ARCHIVE_ORPHAN is recommended only when SCM proves merged+branch-deleted AND no uncommitted changes; every other ambiguous case -> ESCALATE_TO_DECISIONS, never a guessed destructive action.
- Do NOT put git worktree list / gh pr list / git rev-parse inside the scanner/SHA-pin LIBRARY functions. Pass captured text/files in (the fleet_run.validate_manifest pattern) so the libs stay side-effect-free and tests are hermetic. The thin CLI wrapper is the only place that shells out.
- Do NOT implement a real Linear/GitLab tracker adapter under the TRACKER/SCM split item — that item delivers the CONTRACT split in engine.md + template only. And do NOT touch the conflict-aware / NEVER-squash / SHA-pin rules while relabeling; scope strictly to demoting gh from contract to default.
- Do NOT make CONTINUE_WORKER the default or a correctness requirement. SPAWN_WORKER (fresh + ledger-idempotent) stays the default; CONTINUE_WORKER is an OPTIONAL primitive constrained to LIVE-classified rows, aliasing to SPAWN_WORKER for adapters without a resume command. Never re-attach a session whose PR merged or whose branch is gone.
- Do NOT write acceptance tests in the brief's 'mutation must SURVIVE / guards still pass' framing. scripts/mutation_check.py asserts a mutation is CAUGHT (guards FAIL under the mutation); a surviving mutation FAILS the gate. Every mutations.yaml entry here must be one whose guard test goes RED when applied.
- Do NOT treat a deleted-but-merged branch as a SHA-pin FAIL. Post-merge the branch is gone; enforce SHA-pin only for verdict=approve runs whose task is NOT yet MERGED, and skip (N/A) when git rev-parse fails with unknown-revision AND a merged marker exists — otherwise the validator flags every shipped task.
- Do NOT silently merge trace-derived health counts into the markdown-driven dashboard zone counts. The trace stream is sparse/partial by design; render trace health as its OWN labelled 'Run health (from trace.jsonl)' panel so a sparse trace reads as an honest 0, not a contradiction of the ledger.
- Do NOT rely on bwrap/sandbox-exec being universally present for reviewer read-only. Where absent, fall back to a post-exec git-diff/find-newer assertion documented as a best-effort CI/audit gate (detects after the fact), exactly like the existing run-sandboxed RESIDUAL RISK block — do not claim a live OS boundary you cannot guarantee.
